# SPEC: job4j_cars — web application for car sale posts

## 1. Purpose

Build a training web application for publishing car sale posts. The application must follow the job4j Web project style and must be implemented with Spring Boot, Thymeleaf and Hibernate without Spring Data.

This specification is intentionally strict because the project can be returned for rework if Spring Data repositories are used.

## 2. Non-negotiable constraints

1. Do not use Spring Data JPA, `JpaRepository`, `CrudRepository`, `PagingAndSortingRepository`, `@EnableJpaRepositories`, or repository interfaces generated by Spring Data.
2. Use Hibernate through a manually configured Spring bean `SessionFactory`.
3. Inject `SessionFactory` into the persistence layer through constructors.
4. Split application code into three layers:
   - `controller` — HTTP requests and view model preparation;
   - `service` — business rules and authorization checks;
   - `repository` or `store` — Hibernate persistence code.
5. Every main model must have a dedicated repository:
   - `User` -> `UserRepository`;
   - `Post` -> `PostRepository`;
   - `Car` -> `CarRepository`.
6. Avoid inheritance and blind code duplication. Prefer small helper methods inside repositories for transaction/session handling only if they stay simple and readable.
7. README must describe the project and contain interface screenshots.

## 3. Reference project

Use `job4j_todo` as the implementation reference for:

- layer separation;
- manual `SessionFactory` bean;
- repository implementation style;
- Liquibase migration organization;
- Thymeleaf page style;
- README structure.

Do not copy business entities from TODO. Copy only the architectural approach.

## 4. Product scope

### 4.1 Main page

The main page displays a table or card list of all car sale posts.

Required data for each post:

- post id;
- post title;
- car brand;
- car model;
- production year;
- body type;
- price;
- sale status: `Available` or `Sold`;
- author username;
- creation date;
- preview photo if present.

Required actions:

- open post details;
- open create post page;
- for the post author only: change status from available to sold and back.

### 4.2 Create post

The user must be able to create a new car sale post.

Required form fields:

- title;
- description;
- price;
- car brand;
- car model;
- production year;
- body type;
- engine type;
- transmission type;
- mileage;
- photo upload.

The post is created with status `not sold` by default.

### 4.3 Post status

A post has a boolean sale status:

- `sold = false` — car is available;
- `sold = true` — car is sold.

Only the user who created the post can change this status.

### 4.4 Users

Users must exist in the application.

Minimal user fields:

- id;
- name;
- email;
- password.

Authentication may be implemented as simple session-based login/registration, similar to training job4j web projects. Full Spring Security is not required unless already present and simple enough to preserve.

Required user flows:

- registration;
- login;
- logout;
- current user stored in HTTP session;
- anonymous users may view posts;
- only authenticated users may create posts;
- only post author may change post status.

## 5. Domain model

### 5.1 User

Suggested Java class: `ru.job4j.cars.model.User`.

Fields:

- `int id`;
- `String name`;
- `String email`;
- `String password`;
- `LocalDateTime created`.

Constraints:

- email must be unique;
- password can be plain text for this educational task unless the course already requires hashing. Add a TODO in README if plain text is used.

### 5.2 Post

Suggested Java class: `ru.job4j.cars.model.Post`.

Fields:

- `int id`;
- `String title`;
- `String description`;
- `BigDecimal price`;
- `boolean sold`;
- `LocalDateTime created`;
- `User user`;
- `Car car`;
- `Photo photo` or `String photoPath` depending on chosen implementation.

Decision: prefer a separate `Photo` model only if it does not overcomplicate the task. For a first implementation, storing `photoPath` or `photoName` in `Post` is acceptable.

### 5.3 Car

Suggested Java class: `ru.job4j.cars.model.Car`.

Fields:

- `int id`;
- `String brand`;
- `String model`;
- `int year`;
- `BodyType bodyType` or `String bodyType`;
- `EngineType engineType` or `String engineType`;
- `Transmission transmission` or `String transmission`;
- `int mileage`.

For training simplicity, enums are preferred for body, engine and transmission if Thymeleaf forms remain simple. If enums complicate persistence, use dictionary tables.

## 6. Suggested database schema

Tables:

### users

- `id serial primary key`;
- `name varchar not null`;
- `email varchar not null unique`;
- `password varchar not null`;
- `created timestamp not null default now()`.

### cars

- `id serial primary key`;
- `brand varchar not null`;
- `model varchar not null`;
- `year int not null`;
- `body_type varchar not null`;
- `engine_type varchar not null`;
- `transmission varchar not null`;
- `mileage int not null default 0`.

### posts

- `id serial primary key`;
- `title varchar not null`;
- `description text`;
- `price numeric(12, 2) not null`;
- `sold boolean not null default false`;
- `created timestamp not null default now()`;
- `user_id int not null references users(id)`;
- `car_id int not null references cars(id)`;
- `photo_path varchar`.

## 7. Repository requirements

### 7.1 UserRepository

Required methods:

- `Optional<User> findById(int id)`;
- `Optional<User> findByEmailAndPassword(String email, String password)`;
- `Optional<User> findByEmail(String email)`;
- `User create(User user)`.

### 7.2 CarRepository

Required methods:

- `Optional<Car> findById(int id)`;
- `Car create(Car car)`;
- `boolean update(Car car)` if editing is implemented later.

### 7.3 PostRepository

Required methods:

- `List<Post> findAll()` with joined `User` and `Car` to avoid lazy-loading failures in views;
- `Optional<Post> findById(int id)` with joined `User` and `Car`;
- `Post create(Post post)`;
- `boolean updateSoldStatus(int postId, boolean sold)`;
- optional: `List<Post> findByUserId(int userId)`.

All repositories must use Hibernate `SessionFactory`, transactions and HQL/Criteria/native SQL where appropriate.

## 8. Service requirements

### 8.1 UserService

Required methods:

- `Optional<User> register(User user)`;
- `Optional<User> login(String email, String password)`;
- `Optional<User> findById(int id)`.

Business rules:

- do not register duplicate emails;
- return `Optional.empty()` for invalid login.

### 8.2 PostService

Required methods:

- `List<Post> findAll()`;
- `Optional<Post> findById(int id)`;
- `Post create(Post post, User currentUser)`;
- `boolean changeSoldStatus(int postId, boolean sold, User currentUser)`.

Business rules:

- post author is assigned from current session user;
- only post author may change status;
- anonymous user cannot create posts or change statuses.

### 8.3 CarService

Required methods:

- `Car create(Car car)`;
- `Optional<Car> findById(int id)`.

## 9. Controller requirements

### 9.1 PostController

Routes:

- `GET /` or `GET /posts` — list all posts;
- `GET /posts/{id}` — post details;
- `GET /posts/create` — create form;
- `POST /posts/create` — create post;
- `POST /posts/{id}/status` — change sold status.

### 9.2 UserController

Routes:

- `GET /users/register`;
- `POST /users/register`;
- `GET /users/login`;
- `POST /users/login`;
- `GET /users/logout` or `POST /users/logout`.

## 10. View requirements

Templates:

- `posts/list.html`;
- `posts/create.html`;
- `posts/detail.html`;
- `users/login.html`;
- `users/register.html`;
- common fragment for navigation if useful.

UI can be simple Bootstrap or plain CSS. Priority is correctness, not visual polish.

## 11. Photo upload requirements

Minimal acceptable implementation:

- photo is uploaded through `MultipartFile`;
- file is saved to a local directory, for example `files/` or `uploads/`;
- generated filename prevents accidental overwrites;
- relative path or filename is stored in `posts.photo_path`;
- list/detail pages display the image if it exists;
- upload directory is ignored by Git except for `.gitkeep`.

Do not store binary photo data in the database unless the course explicitly requires it.

## 12. Validation and error handling

Minimum validation:

- title is required;
- price must be positive;
- brand and model are required;
- year must be realistic, for example 1900..current year + 1;
- mileage must be non-negative;
- authenticated user is required for create/status actions.

Error handling may be implemented with controller-level checks and simple messages in Thymeleaf.

## 13. Acceptance criteria

The task is complete when:

1. `mvn test` passes.
2. `mvn spring-boot:run` starts the application.
3. No Spring Data dependencies or APIs are used.
4. `SessionFactory` is defined as a Spring bean and injected into repositories.
5. The app has controller, service and repository layers.
6. `UserRepository`, `PostRepository`, and `CarRepository` exist.
7. Anonymous user can view all posts.
8. User can register, login and logout.
9. Authenticated user can create a post with car details and optional photo.
10. Only post author can change sold status.
11. README describes the project and contains screenshots.
